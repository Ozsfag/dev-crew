# П7.4 — Fix `Instant.now()` в тестах

## Контекст

CLAUDE.md запрещает `Instant.now()` в тестах — нужно использовать фиксированный
`Instant.parse("...")`. Причина: `Instant.now()` делает тесты недетерминированными.
При сравнении временных меток (`.isEqualTo()`) тест флакует, потому что между двумя
вызовами `Instant.now()` проходит несколько миллисекунд.

## Проблема

~62 вхождения `Instant.now()` в тестовом коде (19 файлов).

Самые критичные категории:

**1. В helper-методах (создание тестовых объектов):**
```java
// ❌ нестабильно
private TaskModel taskModel() {
  return new TaskModel(..., Instant.now(), Instant.now(), null);
}
```

**2. В `when(timeProvider.now()).thenReturn(Instant.now())` — mock возвращает разное время:**
```java
// ❌ нестабильно
when(timeProvider.now()).thenReturn(Instant.now());
```

**3. В JPA-интеграционных тестах — при создании Entity:**
```java
// ❌ нестабильно
entity.setCreatedAt(Instant.now());
```

## Техническое решение

### Правило замены

| Было | Стало |
|------|-------|
| `Instant.now()` в хелпере | `Instant.parse("2026-01-01T10:00:00Z")` |
| `when(timeProvider.now()).thenReturn(Instant.now())` | `when(timeProvider.now()).thenReturn(NOW)` |
| `Instant.now()` в поле Entity (интеграционный тест) | `Instant.parse("2026-01-01T10:00:00Z")` |

### Паттерн для unit-тестов

```java
@ExtendWith(MockitoExtension.class)
class SomeCommandServiceTest {

  private static final Instant NOW = Instant.parse("2026-01-01T10:00:00Z");  // ← 1 константа

  @Test
  void create_saves_entity_with_timestamp() {
    when(timeProvider.now()).thenReturn(NOW);  // ← мок возвращает константу

    service.create(...);

    var captor = ArgumentCaptor.<SomeModel>captor();
    verify(store).save(captor.capture());
    assertThat(captor.getValue().createdAt()).isEqualTo(NOW);  // ← детерминировано
  }

  private SomeModel someModel() {
    return new SomeModel(UUID.randomUUID(), ..., NOW, NOW);  // ← константа в хелпере
  }
}
```

### Паттерн для интеграционных тестов (JPA)

В JPA-тестах Entity-поля `createdAt/updatedAt` часто заполняются вручную при создании
тестовых данных:

```java
// ❌ было:
entity.setCreatedAt(Instant.now());

// ✅ стало:
private static final Instant NOW = Instant.parse("2026-01-01T10:00:00Z");
entity.setCreatedAt(NOW);
```

Исключение: если `createdAt` заполняется через `@PrePersist` / `@CreationTimestamp` —
не трогать, это правильно. Фикс нужен только там, где значение передаётся явно.

### Стратегия замены

Автоматически через IDE/sed не рекомендуется — нужно проверять каждый контекст.
Последовательность:

1. Для каждого файла: добавить `private static final Instant NOW = Instant.parse("2026-01-01T10:00:00Z");`
   в начало класса (или убедиться, что уже есть)
2. Заменить `Instant.now()` → `NOW` во всех хелперах
3. Заменить `Instant.now()` → `NOW` в `when(timeProvider.now()).thenReturn(...)`
4. Если тест проверяет время «до/после» — использовать `NOW.minusSeconds(...)` / `NOW.plusSeconds(...)`

## Acceptance Criteria

- [ ] `Instant.now()` не встречается ни в одном тестовом файле (можно проверить: `grep -r "Instant.now()" src/test`)
- [ ] Все 19 файлов используют `private static final Instant NOW = Instant.parse(...)`
- [ ] Константа `NOW` объявлена только в хелпер-методах, не inline в asserts
- [ ] `./gradlew test` зелёный после всех замен
- [ ] `./gradlew test -Dspring.profiles.active=tc` зелёный (интеграционные тесты)

## Тест-план

```bash
# Проверить что Instant.now() не осталось
grep -r "Instant.now()" src/test --include="*.java"

# Запустить все тесты
./gradlew test

# Запустить интеграционные
./gradlew test -Dspring.profiles.active=tc
```

## Зависимости

Нет. Это рефакторинг тестов — production-код не затрагивается.
Можно выполнять параллельно с П7.1–П7.3.
